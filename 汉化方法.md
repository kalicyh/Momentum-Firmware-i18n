# 汉化方法

## 历史方案

### `CN_built-in`

`CN_built-in` 的思路是：

- 直接修改源码里的中文文本
- 使用内置字库文件
- 通过 `canvas_set_custom_u8g2_font()` 或类似方式直接切到内置中文字库

这个方案的特点是实现直接，但问题也明显：

- 字库和代码耦合重
- 后续同步上游时冲突多
- 每次增减汉字都要重新维护内置字体源

### `CN`

`CN` 在 `CN_built-in` 基础上继续演进，思路变成：

- 仍然直接修改源码里的中文文本
- 字体不再完全内置，转向资源包方式
- 通过 `asset_packs` 下的字体资源完成中文显示

这个方案比 `CN_built-in` 灵活，但也有问题：

- 中文字体依赖 `asset_packs`
- 需要切换资源包
- 中文主字库和主题资源包系统耦合
- 运行时路径和维护边界不够清晰

## 当前方案

当前 `CN-Clipper` 不再沿用上面两种方法。

现在的方案是：

1. 编译时从中文词条源提取全部中文字符
2. 生成一份总字库 `primary_zh.u8f`
3. 字库作为独立资源输出到 `zh_fonts`
4. 运行时由 `canvas` 私有懒加载 `/ext/zh_fonts/primary_zh.u8f`
5. 遇到中文字符串时临时切换到这份字体进行绘制

默认英文固件不启用这套中文路径。

## 当前实现

### 编译时

中文构建时会读取：

- `localization/zh_CN/strings.json`

并通过：

- `scripts/momentum_zh_font_gen.py`
- `tools/u8g2_cn_tools/bdfconv`
- `tools/u8g2_cn_tools/bdf/wenquanyi_9pt.bdf`

生成：

- `build/<target>/generated/zh_fonts_work/primary_zh.map`
- `build/<target>/generated/zh_fonts_work/primary_zh.c`
- `build/<target>/generated/zh_fonts/primary_zh.u8f`

最终资源分发路径是：

- `/ext/zh_fonts/primary_zh.u8f`

### 运行时

`canvas` 在检测到字符串包含非 ASCII 字节时，会尝试读取：

- `EXT_PATH("zh_fonts/primary_zh.u8f")`

加载成功后临时切换字体，再调用 `u8g2_DrawUTF8()` 完成绘制。

这意味着：

- 不再依赖 `asset_packs/CN-Clipper/Fonts`
- 不再要求手动切换资源包
- 不再需要额外的内置中文字库文件

## 构建方式

### 英文固件

```bash
./fbt COMPACT=1 DEBUG=0
```

或：

```bash
./fbt updater_package COMPACT=1 DEBUG=0
```

### 中文固件

```bash
MOMENTUM_UI_LANG=zh_CN ./fbt COMPACT=1 DEBUG=0
```

或：

```bash
MOMENTUM_UI_LANG=zh_CN ./fbt updater_package COMPACT=1 DEBUG=0
```

只有在 `MOMENTUM_UI_LANG=zh_CN` 时，才会生成并打包 `primary_zh.u8f`。

## 词条维护方式

当前中文字符来源不是扫描整个仓库源码，而是维护一份中文词条源：

- `localization/zh_CN/strings.json`

新增中文时，应优先更新这里，再触发编译生成新字库。

## 当前目录说明

当前正确的中文字体资源目录是：

- `zh_fonts/primary_zh.u8f`
